What Stripe Did for Payments

Draw the parallel: Stripe → payments :: Theatrical → cinema. A technical comparison with code examples.

Research Essay · Theatrical SDK

The Before

In 2010, accepting a credit card payment online looked like this:

You registered for a merchant account. You received API documentation — typically 200+ pages of XML schema definitions. You built a SOAP client, managed card tokenisation manually, handled PCI compliance questionnaires, implemented webhook verification without signature libraries, and wrote retry logic for a gateway that returned different error formats for different failure modes.

It worked. Thousands of businesses processed payments this way. But each integration was a multi-week engineering project, and the result was fragile — tightly coupled to the gateway's idiosyncrasies, difficult to test, and expensive to maintain.

Then Stripe published seven lines of code:

const stripe = require('stripe')('sk_test_...');
const charge = await stripe.charges.create({
  amount: 2000,
  currency: 'usd',
  source: 'tok_visa',
});

The payment gateway underneath was unchanged. The same card networks, the same settlement rails, the same compliance requirements. What changed was the surface area developers interacted with.

The Parallel

Cinema technology today is where payments were in 2009. The platforms are capable. The APIs exist. The documentation is available. What's missing is the developer experience layer.

Compare the experience of integrating cinema booking today vs. what it looks like with Theatrical:

Before: Raw API

// Manually construct auth headers
const token = await fetch('https://auth.moviexchange.com/token', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ apiKey, secret }),
}).then(r => r.json());

// No types, no validation, no retry
const sessions = await fetch(
  `https://api.vista.co/ocapi/v1/sessions?siteId=${siteId}`,
  { headers: { Authorization: `Bearer ${token.access_token}` } }
).then(r => r.json());

After: Theatrical SDK

import { TheatricalClient } from '@theatrical/sdk';

const client = new TheatricalClient({ apiKey: 'your-key' });
const sessions = await client.sessions.list({
  siteId: 'roxy-wellington',
  date: '2026-04-10',
});

Auth handled. Types enforced. Retry built in. Rate limiting proactive. Mock mode available. The API underneath is unchanged — the developer experience layer is what's new.

What Stripe Actually Built

Stripe's genius was not in the seven lines of code. It was in the ecosystem of developer tools around it:

  • Typed clients in every language — not generated stubs, but hand-tuned ergonomic APIs
  • Test mode — realistic fixture data without touching real money
  • CLI — local webhook forwarding, event replay, resource inspection
  • Error taxonomy — predictable error types with actionable messages
  • Idempotency — safe retries built into the protocol

Theatrical builds the same layers for cinema: typed SDKs (TypeScript, C#, Python), mock mode with NZ cinema fixtures, a CLI for code generation and API inspection, a structured error hierarchy with Vista error codes, and proactive rate limiting.
